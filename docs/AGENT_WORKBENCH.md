# Agent Workbench

The agent workbench is the experimental LLM-facing layer of `osrs-ge`.

The core idea is simple: the CLI exposes clean, read-only research primitives.
An LLM can then start from a natural-language query, choose which probes to run,
evaluate the evidence, and iterate without needing a production refactor for
every new alpha idea.

## Principle

The user-authored input should be one plain-English research question.

Examples:

- "Which items are at the bottom of their VWAP range with consistent volume?"
- "Find cheap high-limit items that repeatedly dump and rebound."
- "Show weird low-price food or skilling items with sudden volume spikes today."
- "Find current spreads that are high, liquid, and not just stale quotes."

Everything else should be generated by the agent:

- item universe
- time horizon
- probe family
- volume filters
- recurrence tests
- tax assumptions
- rejection criteria
- follow-up queries

## Commands

Discover available probes:

```bash
osrs-ge agent manifest --json
```

Preflight the local data surface:

```bash
osrs-ge doctor --json
osrs-ge schema --json
```

Run a deterministic multi-probe bundle:

```bash
osrs-ge agent run "items at the bottom of VWAP with consistent volume" --json
```

The output includes:

- derived research spec
- probes run
- raw JSON artifacts per probe
- summary rows
- warnings and next-step suggestions

## Current Probe Families

- `range-bottom`: current price versus estimated VWAP/range, percentile,
  volume consistency, active buckets, and rebound-cycle evidence.
- `patterns`: recent dump/rebound events with tax-adjusted edge and event
  volume.
- `opportunities`: current spread, tax-adjusted margin, ROI, volume, and buy
  limit.
- `movers`: short-term price and volume changes.
- `sql`: read-only local cache inspection.
- `schema`: table, column, row-count, and sync-state discovery.
- `doctor`: cache freshness, API reachability, and setup warnings.

Fixed scanners are probes, not the product boundary. The product boundary is:

1. Read the query.
2. Discover tools.
3. Run several probes.
4. Inspect raw artifacts.
5. Refine thresholds or pull item-specific history.
6. Produce an evidence-backed thesis with caveats.

## Safety Boundary

The workbench is research-only.

- It must not place trades.
- It must not control the game client.
- It must not use account credentials.
- It must not frame outputs as guaranteed profit.

Use language such as "candidate", "setup", "evidence", "risk", "watch rule",
and "manual review".
